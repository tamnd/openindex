# OpenIndex v0.1.0

This is the first tagged release of OpenIndex, a web-scale search engine written in Go whose defining idea is an open, public, content-addressed, signed, and reproducible index. The goal of the project is a search and answer stack that anyone can audit, rebuild byte for byte, and federate, instead of a black box only its operator can see inside.

This release is the foundation. It is not a running search service yet, and it does not pretend to be one. What it is, is the complete set of subsystem seams the engine is built on, each one a clean Go interface with a correct, tested, in-process reference implementation behind it. The heavy production bindings (a real ANN library, an inference server, a distributed object store) are deliberate later swaps behind those same seams. Shipping the seams first is the whole strategy: the shape of the system is fixed, tested, and reviewable before any of the expensive moving parts are bolted on.

## Why a 0.1.0 now

Up to this point the repository had no binary and no tag. Every subsystem landed as a library package across thirteen reviewed pull requests, but there was nothing to install and no released artifact to point at. This release closes that gap in two ways. It introduces a small honest command line binary so there is something real to download and run, and it cuts a versioned release with that binary built for every platform the pure-Go stack supports. From here, the project has a release process that grows with the code rather than one improvised at the end.

## What is in this release

### The binary

`openindex` is a single static binary with no runtime dependencies. At this stage it is an introspection tool, not the engine. It reports the truth about the build and the plan, and nothing more:

- `openindex version` and `openindex -version` print the build version. A released binary carries the tag it was built from; a local `go build` reports `dev`.
- `openindex plan` prints the milestone build sequence, M0 through M9, each with the gate that has to be met before the milestone is considered done.
- `openindex risks` prints the risk register, with the primary risk marked and the recorded mitigations listed under each entry.
- `openindex help` prints usage.

Everything the binary prints is read from the `capacity` package, so the binary and the specification cannot drift apart. If the plan changes in code, the binary's output changes with it.

### The subsystem library

The engine is assembled from one Go package per subsystem, with no `internal` directories and no version-suffixed import paths. Each of the following landed as its own reviewed pull request with its own tests:

- **storage** is the content-addressed storage engine that every other subsystem reads and writes through.
- **crawler** is the polite, consent-aware fetch and discovery layer.
- **index** is the inverted index: the classic text retrieval core.
- **vector** is the semantic index for embedding-based retrieval, behind a seam so the ANN backend is swappable.
- **rank** is ranking and retrieval, including the relevance evaluation harness.
- **serve** is the query serving architecture: sharding, fan-out, hedging, and the good-enough latency cutoff.
- **answer** is the answer engine that composes retrieval with generation, behind a router so the no-LLM, small-model, and frontier paths are all reachable.
- **control** is the control plane that ties crawl and serve together.
- **open** is the federation layer: the signed, content-addressed, reproducible artifact that makes the index open and independently verifiable.
- **telemetry** carries the metric and trace backends, including a RED meter with bounded cardinality and head-sampled fan-out tracing.
- **harness** is the continuous-integration evaluation gate that composes the relevance and answer evaluations into a single pass-or-fail check.
- **capacity** holds the cost and memory models and encodes the build sequence and risk register as data the binary and the tests can both read.

Each package is built against its interface seam with a tested reference implementation. The reference implementations are correct and exercised by tests; they are not stubs. The point is that the seam is the contract, and the production binding is a mechanical replacement that the tests already constrain.

### The release pipeline

This release also ships the workflow that produced it. On a semantic version tag, the release workflow:

1. Resolves the build version once, from the tag.
2. Cross-compiles the binary for seventeen OS and architecture pairs, with CGO disabled so every binary is static and cross-compiles cleanly. The targets are Linux on amd64, arm64, arm, 386, ppc64le, s390x, riscv64, and loong64; macOS on amd64 and arm64; Windows on amd64, arm64, and 386; FreeBSD on amd64 and arm64; and OpenBSD on amd64 and arm64.
3. Verifies each binary's format against its target with `file`, so a Mach-O binary cannot accidentally ship in a Linux archive.
4. Packages each build as a `.tar.gz` (or `.zip` on Windows) alongside a `.sha256` checksum.
5. Runs the binary natively on every target that has a GitHub-hosted runner, asserting that the version flag echoes the build version and that the plan lists the milestones. This proves the binaries start, not merely that they link.
6. Publishes the archives and checksums to this release.

## Downloads and verification

Each platform has an archive and a matching `.sha256` file. After downloading, verify the checksum before extracting:

```
sha256sum -c openindex_v0.1.0_linux_amd64.tar.gz.sha256
```

On macOS use `shasum -a 256 -c`, and on Windows use `CertUtil -hashfile <file> SHA256` and compare against the `.sha256` contents. Then extract and run:

```
tar -xzf openindex_v0.1.0_linux_amd64.tar.gz
./openindex_v0.1.0_linux_amd64/openindex version
```

macOS binaries are not notarized in this release. If Gatekeeper blocks the binary, clear the quarantine attribute with `xattr -d com.apple.quarantine ./openindex` after you have verified the checksum.

## What this release is not

It is not a running search engine. There is no server to start, no index to query, and no answer endpoint to call. Those paths exist as library seams with reference implementations, but they are not wired into a runnable service, because their production form is the next phase of work. The binary in this release intentionally exposes only what the library can answer truthfully today. A command that claimed to serve queries while doing nothing real would be worse than no command at all.

## Known limitations

- The reference implementations are correct and tested, but they are in-process and not tuned for scale. They exist to pin behavior, not to carry production load.
- The production bindings (ANN backend, inference server, distributed storage) are not present. The seams that will hold them are.
- The macOS binaries are unsigned and unnotarized.

## What comes next

The build sequence the binary prints under `openindex plan` is the roadmap. M0 is a single-node searchable index measured against an Elasticsearch baseline on a Common Crawl slice. From there the milestones add distributed serving, the open verifiable artifact, real ranking, the hybrid vector index, the answer engine, the behavioral flywheel, crawling at scale, federation, and finally scale-out. Each milestone has a concrete gate, and each gate is something a test or a benchmark can decide. The seams shipped in this release are what make that sequence a series of mechanical swaps rather than rewrites.

## Thanks

This release is the result of thirteen focused pull requests, each building one subsystem against its seam, plus the binary and release pipeline added here. The discipline of building against interfaces with tested reference implementations is what lets a project this size stay reviewable from the first commit.
